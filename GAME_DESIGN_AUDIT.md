## 二、核心游戏循环审查

### 2.1 设计支柱推断

从代码和文档中，可推断出 3 个隐含设计支柱：

| # | 支柱 | 代码支撑 | 评价 |
|---|------|----------|------|
| 1 | **在游戏中学习** | 答题=战斗，错题→AI辅导 | ✅ 核心 fantasy 成立 |
| 2 | **持续成长感** | 经验/等级/成就/星级/排行榜 | ✅ 多维度正反馈 |
| 3 | **轻松田园氛围** | Sprout Lands素材/像素风/NPC奶牛 | ✅ 视觉统一 |

**🔴 缺失的支柱**: 没有明确的「社交/竞争」支柱。排行榜是只读展示，缺少好友对战、合作闯关等社交驱动力。对于教育游戏，这意味着缺少外部留存钩子。

## 三、可玩性深度审查

### 3.1 核心动词分析

| 动词 | 实现 | 频率 | 满意度 |
|------|------|------|--------|
| **移动** | WASD/方向键 在 960×640 地图上 | 高 | ⚠️ 地图30×20格≈960×640px，活动空间小，3秒就能走完对角线 |
| **答题** | 4选1选择题 | 极高 | 🔴 **题型单一是最大可玩性瓶颈** |
| **躲避** | Boss子弹/冲锋 | 低 | ✅ ChargingBoss的警告→闪避是最有趣的交互 |
| **对话** | AI NPC聊天 | 低 | ✅ 答错触发的学习辅导流程设计得当 |
| **收集** | 星级/成就/经验 | 中 | ⚠️ 纯数字累积，缺少可见的收集物（如词卡图鉴） |

### 3.2 🔴 题型单一问题（最大可玩性缺陷）

**现状**: 
- 前端只实现了 `choice_en2cn`（英→中4选1）一种题型
- 自适应引擎定义了 5 级难度对应 5 种题型（拼写、填空、翻译等），但**前端完全没有实现 2-5 级的 UI**
- `QuizModal.vue` 和 `BossQuizModal.vue` 都只渲染4个按钮选项

**后果**:
- 不管自适应引擎返回什么难度，玩家永远做相同形式的题——这会让30关200+次答题变得极度单调
- 学习效果打折：选择题只考察"认知"，缺少"回忆"和"产出"层次的练习

**建议**: 至少实现拼写题（填空输入框）和中→英反向选择题，用不同 UI 组件渲染。

### 3.3 地图重复性问题

**现状**:
- 所有30关使用**完全相同的地图生成逻辑**（30×20的围栏 + 40个随机装饰物）
- 怪物/NPC/Boss的位置是随机生成的，但地形布局无变化
- 无障碍物、无通道、无地形差异

**后果**:
- 第1关和第30关的地图体验完全一样——玩家到第5关就会产生审美疲劳
- Boss的空间策略（冲锋/子弹躲避）在空旷地图上几乎无意义——到处都能跑

**建议**: 
- 至少准备3-4套地图模板（加入水域/山丘/桥梁障碍物）
- 后期章节应有封闭走廊、Boss竞技场等差异化布局

### 3.4 Boss 系统评价

Boss 系统是整个项目**最有设计感的部分**，三种类型各有特色：

| Boss | 设计意图 | 实际体验 | 问题 |
|------|----------|----------|------|
| RoamingBoss | 追击→接触→答题 | ✅ 有"狩猎"的感觉 | 速度30-45太慢，玩家160速度秒追上 |
| TurretBoss | 闪避子弹→接近→答题 | ✅ 增加了动作维度 | 子弹4秒存活+120-150速度=几乎不可能被击中 |
| ChargingBoss | 识别警告→闪避→反击 | ⭐ **最佳设计** | 冲锋200-250速度+警告闪烁是优秀的risk/reward设计 |

**关键问题**: Boss 战的"答题"部分和普通怪完全一样（4选1），Boss 应该有更具仪式感的答题体验（如多轮连续答、限时更短、错误惩罚更重）。目前 BossQuizModal 已经支持多轮答题，但 `questionsNeeded` 固定为 1，实际上每次接触只答一题，和普通怪没区别。

## 四、经济系统与数值平衡审查

### 4.1 货币/资源流

```
┌──────────────────────────────────────────────────────────────┐
│                       资源来源 (Sources)                      │
│                                                               │
│  答题得分: 100-550分/题 (基础+连击+时间+难度倍率)              │
│  Boss击败: 500×倍率 = 400-750分/Boss                          │
│  每日签到: 10+min(streak×5, 50) = 15-60经验/天                │
│                                                               │
│  经验来源: totalExp += floor(scoreDelta/2)                     │
│  等级: level = floor(totalExp/100) + 1                        │
└──────────────────────────────┬────────────────────────────────┘
                               │
                               ▼
┌──────────────────────────────────────────────────────────────┐
│                       资源消耗 (Sinks)                        │
│                                                               │
│                      ⚠️ 完全没有消耗机制                      │
│                                                               │
│  无商店购买、无升级消耗、无重开费用、无道具消耗                 │
│  分数和经验只增不减                                            │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 🔴 经济通胀风险

**问题**: 游戏经济是纯 source 无 sink 的单向通胀模型。

- 分数和经验永远只增不减（服务端 `scoreDelta = Math.max(0, score - oldScore)` 确保重玩不扣分）
- 但这也意味着**重玩相同关卡只要打出更高分就会持续获得经验**
- 没有任何消耗机制来平衡资源流入

**实际影响**:
- 等级 = `floor(totalExp/100) + 1`，每100经验升一级
- 一关典型得分约1500-3000分 → 经验750-1500 → 可能一关升7-15级
- 30关全通后，玩家等级可能达到 200+，但等级完全无意义（不解锁任何东西）

**结论**: 经济系统本质上只是一个累计计数器，不构成真正的"经济"。对于毕设项目这可以接受，但如果要增加深度，需要引入消耗机制（如解锁角色皮肤需要金币、使用提示需要消耗资源等）。

### 4.3 计分公式审查

```
答题得分 = (100 × 难度等级 + min(combo×10, 50) + timeBonus) × difficultyMultiplier

timeBonus (客户端 helpers.js):   <5s → +30,  <10s → +15
timeBonus (服务端 scoringService): <3s → +50,  <5s → +30,  <10s → +15
```

**🔴 客户端与服务端计分公式不一致！**

| 参数 | 客户端 (`helpers.js`) | 服务端 (`scoringService.js`) |
|------|----------------------|---------------------------|
| 时间加成<3秒 | 不存在（归入<5秒=30） | +50 |
| 时间加成<5秒 | +30 | +30 |
| 使用提示扣分 | 不存在 | baseScore × 0.5 |
| 难度倍率 | 在 `GameView.vue` 额外乘 | 服务端不乘 |

**实际使用的是客户端公式**（`GameView.vue` 调用 `calculateScore` from `helpers.js`），服务端的 `scoringService` 虽然被 import 但**从未在路由中被调用**——它只是一个未连接的服务模块。

### 4.4 星级评定不一致

| 条件 | 客户端 (`LevelManager.js`) | 服务端 (`scoringService.js`) |
|------|---------------------------|----------------------------|
| 3星满血 | `lives === difficultyConfig.lives`（按难度） | `livesRemaining === 3`（硬编码3） |
| 2星生命 | `lives >= ceil(maxLives/2)`（动态） | `livesRemaining >= 2`（硬编码） |

**实际使用的是客户端逻辑**。服务端的星级评定函数同样是未连接的死代码。

### 4.5 进度曲线分析

```
章节难度 ─────────────────────────────────────
          Ch1     Ch2     Ch3     Ch4     Ch5     Ch6
          基础    自然    商业    学术    易混    综合
词汇数:   80      70      70      70      50      60
Boss HP:  3-4     3-4     3-4     4-5     4-5     4-6
词汇难度: 1-2     1-3     2-3     3-4     3-5     2-5
怪物数:   16      14      14      14      10      12
```

**问题**:
1. **Ch5 暗影深渊（易混词）的词汇数最少（50词）**，但难度最高（3-5）→ 玩家在最难的章节反而战斗次数最少，这违反了"高难度需要更多练习机会"的学习原则
2. **Ch6 综合复习没有新词**，理论上应该从 Ch1-5 抽取词汇，但代码里 `category: 'review_basic'` 等标签是独立存储的，需要确认词库中是否真的有这些 category 的词
3. **Boss HP 的递增过于温和**：Ch1 的 3HP 到 Ch6 的 6HP 只差一倍，考虑到每次接触只答1题，即使最终Boss也只需接触6次

## 五、玩家心理与动机审查

### 5.1 动机框架分析 (Bartle 模型)

| 玩家类型 | 满足度 | 分析 |
|----------|--------|------|
| **成就者** (Achiever) | ⭐⭐⭐⭐ | 16成就+3星评级+排行榜，满足度高 |
| **探索者** (Explorer) | ⭐⭐☆☆ | 地图无秘密，无隐藏内容，无可探索的分支路径 |
| **社交者** (Socializer) | ⭐☆☆☆ | 仅有只读排行榜，无好友系统/组队/分享 |
| **杀手** (Killer) | ⭐☆☆☆ | 无PvP/竞争模式 |

### 5.2 情感曲线设计

```
    兴奋度
    ↑
    │         Boss战        Boss战        Boss战
    │          /\           /\            /\          ← 章节Boss是情感高潮点
    │    怪物 /  \   怪物  /  \    怪物  /  \
    │     /\/    \/\/     /    \/\/     /    \
    │    /                                    \
    │   / 新手引导                              \  ← 通关后断崖式下跌
    │  /                                        \_____ 无终局内容
    └──────────────────────────────────────────────── 时间 →
       Ch1          Ch2-Ch4          Ch5-Ch6    通关后
```

**优点**:
- ✅ 答错后自动触发AI辅导（"小智"），将挫败感转化为学习机会——**情感设计最佳的环节**
- ✅ 答错怪物变半透明→2秒后恢复→强制重新挑战，符合"间隔重复"学习原则
- ✅ 连击系统（combo）+ 即时分数飘字提供了微观正反馈
- ✅ 成就弹窗的缩放动画效果好

**问题**:
- 🔴 **死亡螺旋**：答错→扣1命→连击清零→接下来的题目不会变简单（自适应难度是基于最近20题的整体表现，不是即时调整）→如果连续答错2-3题就可能 Game Over，而 easy 模式4条命、hard 模式只有2条命
- ⚠️ **"惩罚>奖励"感知失衡**: 答错扣命+清零连击+怪物恢复+触发对话面板（被迫看辅导），感觉像被罚了4次；答对只得分+怪物消失+金币特效，感觉只奖了2次

### 5.3 留存机制审查

| 机制 | 存在 | 有效性 | 说明 |
|------|:----:|--------|------|
| 每日签到 | ✅ | ⚠️ | 只给10-60经验，经验又无用处 |
| 连续登录 | ✅ | ⚠️ | 断签直接归1，惩罚过重 |
| 每日任务 | ❌ | - | 无"今日答题X道"之类的每日目标 |
| 赛季/周期性 | ❌ | - | 无赛季排行榜/限时活动 |
| 社交绑定 | ❌ | - | 无好友/分享/对战 |
| 内容更新 | ❌ | - | 静态400词，无DLC/词包扩展机制 |

## 六、新手引导（Onboarding）审查

### 6.1 当前引导流程

```
登录 → 进入/game → Phaser Boot(进度条) → MenuScene(菜单)
  → 点"开始冒险" → LevelSelect(选关) → 选择第1章第1关
  → GameIntro(5页操作指引) → 进入 WorldScene
```

### 6.2 GameIntro 内容审查

| 页 | 内容 | 评价 |
|----|------|------|
| 1 | 移动操作（WASD/方向键） | ✅ 必要信息 |
| 2 | 普通怪物（小鸡）的答题规则 | ✅ 但纯文字，缺交互 |
| 3 | Boss怪物（3种类型说明） | ⚠️ 信息量过大，一次性展示3种Boss |
| 4 | NPC小智 + 每日签到 | ✅ |
| 5 | 星级/成就概览 | ⚠️ 在还没玩之前告诉星级规则为时过早 |

### 6.3 🟡 新手引导问题

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| 1 | **纯文字说明，无交互式教程** | 🟡 | 5页PPT式的"说明书"，玩家很可能跳过不看 |
| 2 | **首关无保障成功设计** | 🟠 | Ch1-Lv1 就有Boss + 限时30秒 + 只有3命，新手可能首关就Game Over |
| 3 | **一次性信息轰炸** | 🟡 | Boss类型、成就系统、AI助手全在第一次进入时展示 |
| 4 | **`skipIntro` 用 localStorage** | 🔵 | 只有"不再显示"选项，换浏览器/清缓存后会重新弹出 |

### 6.4 新手引导对照检查清单

- [x] 核心动词在首次控制后30秒内引入（WASD移动✅）
- [ ] 🔴 第一次成功有保障——**教程第一环节不可能失败**（当前可能首关就死）
- [ ] 🔴 每个新机制在安全、低风险的环境中引入（Boss从Ch1-Lv1就出现）
- [ ] 🟡 玩家通过探索（而非文本）发现至少一个机制（全部是文字说明）
- [x] 第一次会话以钩子结束——关卡结算+星级+经验动画✅

### 7.2 🟡 系统断连问题

1. **Boss击败不触发成就检查**: `WorldScene.onBossQuizResult` 中 Boss 被击败后只加了分数，没有调用 `scoreSystem.checkAchievements()`。如果一个成就需要"击败所有Boss"，这条路径不通（虽然当前16个成就中没有这个）

2. **AI对话次数不追踪**: `achievementContext.npcChats` 需要 ≥10 触发"小智的朋友"成就，但代码中**从未对 `npcChats` 进行自增操作**。`closeChatPanel()` 和 `openManualChat()` 都没有 `achievementContext.npcChats++`

3. **登录连续天数成就仅依赖 `achievementContext.loginStreak`**，但这个值**从未从服务端同步**（`achievementContext` 是纯前端 reactive 对象，初始化全为0）。这意味着 `login_3/7/30` 三个登录成就**永远不会触发**

4. **自适应难度返回值被丢弃**: `submitQuizRecord` 的响应中包含 `adaptiveDifficulty`，但 `GameView.vue` 的 `.catch()` 吞掉了返回值，完全没有使用自适应结果来调整下一题难度

- **每关10-16词，但怪物数也是6-15只**——词汇和怪物接近1:1映射

- 但反过来，如果词汇多于怪物（16词但只有8只怪），多出的词完全不会出现
- 没有"错词本→定向复习"的关卡模式

## 九、技术-设计衔接问题

### 9.1 🔴 严重的设计-实现脱节

| # | 设计意图 | 实际实现 | 影响 |
|---|----------|----------|------|
| 1 | 5级自适应难度（选择→拼写→翻译） | 只实现了选择题 | 核心学习机制缺失 |
| 2 | Boss需多轮连续答题击败 | `questionsNeeded`固定为1 | Boss战没有压迫感 |
| 3 | 服务端积分/星级服务 | 前端自算，服务端代码是死代码 | 计分可被客户端篡改 |
| 4 | 16种成就全部可触发 | 3个登录成就+1个NPC成就永不触发 | 25%成就是死的 |

### 9.2 🟡 音效完全缺失

- `client/public/assets/audio/` 目录为空
- 代码中**没有任何音频播放逻辑**
- 对于像素风游戏，音效是"手感"的关键组成部分——答对的叮咚、答错的嗡嗡、Boss出现的重低音、连击的递进音阶等
- **这是可玩性损失最大的单一缺陷之一**

### 9.3 🟡 BattleScene 未使用

- `BattleScene.js` 在 Phaser config 中注册，但**从未被任何场景调用 `scene.start('BattleScene')`**
- 它的 `shutdown()` 方法用 `eventBus.off(EVENTS.QUIZ_ANSWERED)` 无参数移除整个事件，会影响其他监听者
- 建议：要么删除这个废弃场景，要么将Boss战的视觉表现移到这里

## 十二、最终结论

### 做得好的地方 ✅

1. **Vue + Phaser 的混合架构设计出色** — EventBus 解耦、LevelManager 单一数据源、场景 shutdown 清理（修复了之前的内存泄漏）
2. **AI 学伴"小智"的触发设计** — 答错自动弹出辅导→SSE流式对话→快捷按钮（记忆技巧/例句/词根），这是整个项目最有教育价值的设计
3. **Boss 多态设计** — 3种Boss类型+工厂模式+独立AI行为，是整个游戏最有"游戏设计"含量的部分
4. **田园像素风视觉统一性** — 从登录页到游戏内再到结算页，颜色主题（#5b8c3e/#ffc847/#8b6914）+字体风格一以贯之
5. **数据分析仪表盘** — 学习热力图/章节正确率雷达图/易错词Top10，对教育产品很重要

### 需要改进的地方 🔴

1. **题型单一是致命伤** — 只有4选1选择题，500次答题体验完全相同
2. **25%成就是死代码** — 3个登录成就+1个NPC成就因为数据不同步永远无法解锁
3. **经济系统是空壳** — 分数/经验只增不减，等级无任何游戏效果
4. **音效完全缺失** — 对像素风游戏来说几乎是灾难性的体验缺失
5. **地图零差异化** — 30关同一张地图，探索维度为零

### 一句话总结

> 🌿 **「词汇大冒险」在系统架构和教育理念上是一个扎实的毕设项目，核心"战斗即学习"的fantasy成立，AI辅导流程设计优秀。但作为"游戏"，它在最关键的即时循环上缺乏变化和手感——题型单一、地图重复、音效缺失——这三个问题共同导致了"30分钟后不想再玩"的可预见结果。好消息是，这些都是可以通过2-3天的迭代显著改善的。**
